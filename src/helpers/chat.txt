# Citizen–Notary Blockchain Notarization System Summary

## Tech Stack

* Node.js
* Express
* Prisma ORM
* PostgreSQL
* Wallet-based authentication
* JWT authorization

---

# Core Actors

## Citizen (`CITIZEN`)

* Creates transactions
* Can edit or cancel transactions before notary claim
* May be a signer if explicitly added as one
* Creator is NOT automatically a signer

## Notary (`NOTARY`)

* Claims transactions
* Reviews transaction data
* Approves or rejects transactions
* Signs after all signers have completed signing
* Is NOT considered a legal signer

## Admin (`ADMIN`)

* Reserved for future system administration

---

# Authentication Flow

1. Wallet requests login
2. Server generates nonce
3. Wallet signs nonce
4. Signature is verified
5. JWT issued

JWT payload:

```json
{
  "userId": "...",
  "role": "CITIZEN | NOTARY | ADMIN"
}
```

## Transaction

Represents a legal workflow.

Fields:

* id
* templateId
* body (JSON)
* creatorId
* notaryId
* status

Blockchain fields:

* chainId
* contractAddress
* txHash

---

## TransactionSigner

Represents legal participants.

Fields:

* transactionId
* walletId
* role
* signature
* signedAt

Constraints:

* unique(transactionId, walletId)

Rules:

* Creator ≠ signer
* Notary ≠ signer
* Legal parties are defined only through TransactionSigner

---

## Nonce

Used for wallet authentication.

Fields:

* nonce
* address
* expiresAt

# Transaction Statuses

Current recommended state machine:

```text
DRAFT
↓
PENDING_NOTARY
↓
CLAIMED_BY_NOTARY
↓
WAITING_FOR_SIGNERS
↓
WAITING_FOR_NOTARY_SIGNATURE
↓
SUBMITTED
↓
CONFIRMED
```

Terminal states:

```text
REJECTED
FAILED
CANCELLED
```

---

# Transaction Creation Flow

Input:

```json
{
  "templateId": "...",
  "body": {},
  "signers": [
    {
      "walletId": "...",
      "role": "BUYER"
    }
  ]
}
```

Validation:

* templateId required
* body required
* signers array required
* signers not empty
* no duplicate walletIds
* wallets must exist
* wallets must be active
* wallet owners must be active

Creation:

* Create Transaction
* Create nested TransactionSigner records
* Status = PENDING_NOTARY

---

# Business Rules

* Creator is not automatically a signer
* Notary is not a signer
* Wallet must be active
* User must be active
* Signer roles are template-driven
* Only TransactionSigner determines legal participation

## Get Transactions Where User Is A Signer

Endpoint:

```http
GET /transactions/signing
```

Optional filter:

```http
GET /transactions/signing?status=WAITING_FOR_SIGNERS
```

Logic:

* User owns wallet
* Wallet exists in TransactionSigner
* Optional status filter

---

## Get Transactions Created By User

Endpoint:

```http
GET /transactions/created
```

Optional filter:

```http
GET /transactions/created?status=CONFIRMED
```

Logic:

```text
Transaction.creatorId = currentUser.id
```

# Recommended Next Features

## Notary Claim

Endpoint:

```http
POST /transactions/:id/claim
```

Requirements:

* Only notaries can claim
* Atomic update
* Prevent multiple notaries claiming same transaction

Transition:

```text
PENDING_NOTARY
→
CLAIMED_BY_NOTARY
```

---

## Notary Approval

Transition:

```text
CLAIMED_BY_NOTARY
→
WAITING_FOR_SIGNERS
```

---

## Sign Transaction

Signer submits wallet signature.

Updates:

```text
TransactionSigner.signature
TransactionSigner.signedAt
```

When all signers signed:

```text
WAITING_FOR_SIGNERS
→
WAITING_FOR_NOTARY_SIGNATURE
```

---

## Notary Final Signature

Transition:

```text
WAITING_FOR_NOTARY_SIGNATURE
→
SUBMITTED
```

---

## Blockchain Confirmation Worker

Transition:

```text
SUBMITTED
→
CONFIRMED
```

after on-chain confirmation.
